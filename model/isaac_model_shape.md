## Robot Dog Isaac Lab Model: Desired Movement Vector and Input Shape

### 1. Where the desired movement vector comes from

- **High-level concept**
  - The “desired movement vector” for the robot dog in the velocity-flat task is the **base velocity command** in the robot’s base frame, generated by a command term named `base_velocity`.
  - This command is produced by a command generator of type `UniformVelocityCommand` configured through `UniformVelocityCommandCfg`.
  - For each simulated environment, the command is a 3D vector:
    - $v_x$ – desired linear velocity along the robot’s forward $x$-axis (m/s)
    - $v_y$ – desired linear velocity along the robot’s lateral $y$-axis (m/s)
    - $\omega_z$ – desired angular yaw rate about the vertical $z$-axis (rad/s)

- **Configuration location**
  - The generic locomotion velocity configuration defines the command spec:


  - The robot dog velocity-flat configuration **inherits** this and then **overrides the ranges and some flags** to match the small “chihuahua-sized” robot:

  - Key implications for Robot Dog:
    - **Sampling ranges**
      - $v_x \sim \mathcal{U}(-0.6, 0.6)\,\text{m/s}$
      - $v_y \sim \mathcal{U}(-0.6, 0.6)\,\text{m/s}$
      - $\omega_z \sim \mathcal{U}(-0.8, 0.8)\,\text{rad/s}$
    - `heading_command = False` means the yaw command is **sampled directly as an angular velocity**, not via heading error control.
    - `rel_standing_envs` and `rel_heading_envs` control probabilities of “standing still” and “heading-based” envs; for Robot Dog, it enforces no heading-command-based yaw.

```30:88:source/isaaclab/isaaclab/envs/mdp/commands/velocity_command.py
class UniformVelocityCommand(CommandTerm):
    r"""Command generator that generates a velocity command in SE(2) from uniform distribution.
    ...
    """

    cfg: UniformVelocityCommandCfg

    def __init__(self, cfg: UniformVelocityCommandCfg, env: ManagerBasedEnv):
        ...
        # obtain the robot asset
        self.robot: Articulation = env.scene[cfg.asset_name]

        # create buffers to store the command
        # -- command: x vel, y vel, yaw vel, heading
        self.vel_command_b = torch.zeros(self.num_envs, 3, device=self.device)
        self.heading_target = torch.zeros(self.num_envs, device=self.device)
        self.is_heading_env = torch.zeros(self.num_envs, dtype=torch.bool, device=self.device)
        self.is_standing_env = torch.zeros_like(self.is_heading_env)
        # -- metrics
```

- **Shape and semantics**
  - `self.vel_command_b` is the core **desired movement vector**:
    - Shape: `(num_envs, 3)`
    - Coordinates: robot base frame (“b” suffix).
    - Columns:
      - `[:, 0]` → desired linear velocity along forward $x$ (m/s).
      - `[:, 1]` → desired linear velocity along lateral $y$ (m/s).
      - `[:, 2]` → desired angular yaw rate around $z$ (rad/s).

- **How sampling is done**
  - When the command term is resampled (based on `resampling_time_range`), the `_resample_command` method draws new commands from the configured ranges:

```126:141:source/isaaclab/isaaclab/envs/mdp/commands/velocity_command.py
    def _resample_command(self, env_ids: Sequence[int]):
        # sample velocity commands
        r = torch.empty(len(env_ids), device=self.device)
        # -- linear velocity - x direction
        self.vel_command_b[env_ids, 0] = r.uniform_(*self.cfg.ranges.lin_vel_x)
        # -- linear velocity - y direction
        self.vel_command_b[env_ids, 1] = r.uniform_(*self.cfg.ranges.lin_vel_y)
        # -- ang vel yaw - rotation around z
        self.vel_command_b[env_ids, 2] = r.uniform_(*self.cfg.ranges.ang_vel_z)
        # heading target
        if self.cfg.heading_command:
            self.heading_target[env_ids] = r.uniform_(*self.cfg.ranges.heading)
            # update heading envs
            self.is_heading_env[env_ids] = r.uniform_(0.0, 1.0) <= self.cfg.rel_heading_envs
        # update standing envs
        self.is_standing_env[env_ids] = r.uniform_(0.0, 1.0) <= self.cfg.rel_standing_envs
```

  - After raw sampling, `_update_command` may modify the vector:

```144:163:source/isaaclab/isaaclab/envs/mdp/commands/velocity_command.py
    def _update_command(self):
        """Post-processes the velocity command.
        ...
        """
        # Compute angular velocity from heading direction
        if self.cfg.heading_command:
            ...
            heading_error = math_utils.wrap_to_pi(self.heading_target[env_ids] - self.robot.data.heading_w[env_ids])

```

  - For **Robot Dog flat**:
    - `heading_command` is explicitly set to `False`, so:
      - No conversion from heading error to $\omega_z$.
      - Angular component is **purely uniform** between `(-0.8, 0.8)`.
    - `is_standing_env` can still zero-out the whole vector for some envs, depending on `rel_standing_envs`.

  - The observation manager computes each group by:

    - Optionally storing history and flattening it.

### 4. How the movement vector ultimately reaches the model

- **Env step ordering and command update**
  - Robot Dog uses `RobotDogVelocityFlatEnv`, which subclasses `ManagerBasedRLEnv` and mainly adds action caching and collision fixes:


  - The **base** `ManagerBasedRLEnv.step` function performs, in order:
    1. Processes the action and applies it to the sim.
    2. Advances physics for `decimation` steps.
    3. Computes rewards and terminations.
    4. Resets terminated envs.
    5. **Updates commands** via the command manager.
    6. Applies any interval events.
    7. **Computes observations** via the observation manager and returns them.

```153:241:source/isaaclab/isaaclab/envs/manager_based_rl_env.py
        self.obs_buf = self.observation_manager.compute(update_history=True)

```

  - That means:
    - The command generator is advanced **once per env step** in sync with the RL step.
    - The observation `velocity_commands` always reflects the **most recent desired base velocity** at the time the obs is returned to the policy.

  - Isaac Lab’s RSL-RL integration uses `ManagerBasedRLEnv` as a Gym-style vectorized env whose **observation space** is a dictionary keyed by observation group (here `{"policy": <Box>}`).
  - For Robot Dog, the PPO runner config is:

  - Internally, the RSL-RL runner flattens the `"policy"` observation group into a 2D tensor of shape:
    - `(num_envs, obs_dim)`
  - Here, `obs_dim` is the sum of the dimensions of all enabled `policy` terms:
    - $3$ (base_lin_vel) + $3$ (base_ang_vel) + $3$ (projected_gravity) + $3$ (**velocity_commands**) + joint-related dims + action dims.
  - The **desired movement vector** is simply **three contiguous components** in this flattened policy observation:
    - Indices immediately after the base state terms (exact indices depend on joint and action sizes, but relative order is fixed by the `PolicyCfg` definition).

- **Flow summary to the model**
  1. **Sampling**:
     - At command resampling times, `UniformVelocityCommand` samples $(v_x, v_y, \omega_z)$ from the configured uniform ranges, zeroing them for some “standing” envs.
  2. **Storage**:
     - The sampled vectors are stored in `vel_command_b` with shape `(num_envs, 3)` in base frame.
  3. **Command manager access**:
     - `env.command_manager.get_command("base_velocity")` returns `vel_command_b`.
  4. **Observation term**:
     - `generated_commands(env, command_name="base_velocity")` returns this tensor to the observation manager as `velocity_commands`.
  5. **Observation pipeline**:
     - The observation manager optionally adds noise/clip/scale (none for this term), then concatenates `velocity_commands` with other policy observations, yielding a `(num_envs, obs_dim)` policy observation tensor.
  6. **RL interface**:
     - `ManagerBasedRLEnv.step()` returns `obs["policy"]` (this concatenated tensor) to the PPO runner.
  7. **Model input**:
     - The PPO actor and critic networks take this flattened observation as input; the three components of the desired movement vector act as the **command-conditioning input** that tells the policy what base motion to realize.

### 5. Mental model: what the policy “sees” and how it uses it

- **From the model’s perspective**
  - At each step, for each env, the policy receives a 1D observation vector containing:
    - Current base linear and angular velocities (what the robot is currently doing).
    - Projected gravity (orientation-relative gravity vector).
    - **Desired base velocity command** $(v_x^{\text{cmd}}, v_y^{\text{cmd}}, \omega_z^{\text{cmd}})$.
    - Joint positions and velocities (pose of all legs).
    - Last action (for smoothness/regularization).
  - The policy learns a mapping:
    - $(\text{state}, \text{command}) \rightarrow \text{joint position offsets}$ (the action) that should make the real base velocity follow the command.

### 6. Concise end-to-end data path

- **End-to-end chain for the desired movement vector:**
  1. `RobotDogVelocityFlatEnvCfg` configures `commands.base_velocity` ranges and disables heading commands.
  2. `UniformVelocityCommandCfg` instantiates `UniformVelocityCommand`, which stores the desired base velocity as `vel_command_b ∈ ℝ^{num_envs×3}`.
  3. On each env step, `command_manager.compute(dt)` updates this command, resampling it when needed.
  4. The observation term `velocity_commands = ObsTerm(func=mdp.generated_commands, params={"command_name": "base_velocity"})` calls `generated_commands`, which returns `env.command_manager.get_command("base_velocity")` = `vel_command_b`.
  5. The observation manager concatenates `velocity_commands` with other policy observations to form `obs["policy"] ∈ ℝ^{num_envs×obs_dim}`.
  6. The RSL-RL PPO runner flattens `obs["policy"]` per env into a 1D feature vector and feeds it into the actor and critic MLPs; the three components of the desired movement vector are fixed in position in this feature vector and condition the policy on the commanded base motion.***
# Isaac Lab Model Shape - Complete Specification

## 48-Dimensional Observation Vector (Input to Model)

### Index 0-2: `base_lin_vel` (3 values) - Linear Velocity in World Frame
- **Units**: meters/second (m/s)
- **Frame**: World frame (global coordinate system)
- **Description**: Linear velocity of the robot's base/body center in world coordinates

**Index 0**: `base_lin_vel[0]` = `vx` (forward/backward velocity in world X)
- Positive: Moving forward in world X
- Negative: Moving backward in world X
- Current implementation: Always `0.0` (no odometry available)

**Index 1**: `base_lin_vel[1]` = `vy` (left/right velocity in world Y)
- Positive: Moving left in world Y
- Negative: Moving right in world Y
- Current implementation: Always `0.0` (no odometry available)

**Index 2**: `base_lin_vel[2]` = `vz` (up/down velocity in world Z)
- Positive: Moving up
- Negative: Moving down
- Current implementation: Always `0.0` (no odometry available)

---

### Index 3-5: `base_ang_vel` (3 values) - Angular Velocity in World Frame
- **Units**: radians/second (rad/s)
- **Frame**: World frame
- **Description**: Angular velocity (rotation rate) of the robot's base around world axes

**Index 3**: `base_ang_vel[0]` = `wx` (angular velocity around world X-axis)
- Positive: Rotating forward (pitch up)
- Negative: Rotating backward (pitch down)
- Source: Your gyroscope `roll` converted from deg/s to rad/s
- Conversion: `roll_deg_per_sec × (π/180) = wx`

**Index 4**: `base_ang_vel[1]` = `wy` (angular velocity around world Y-axis)
- Positive: Rotating left (roll left)
- Negative: Rotating right (roll right)
- Source: Your gyroscope `pitch` converted from deg/s to rad/s
- Conversion: `pitch_deg_per_sec × (π/180) = wy`

**Index 5**: `base_ang_vel[2]` = `wz` (angular velocity around world Z-axis)
- Positive: Rotating counterclockwise (yaw left)
- Negative: Rotating clockwise (yaw right)
- Source: Your gyroscope `yaw` converted from deg/s to rad/s
- Conversion: `yaw_deg_per_sec × (π/180) = wz`

---

### Index 6-8: `projected_gravity` (3 values) - Gravity Direction in Body Frame
- **Units**: Unitless (normalized unit vector)
- **Frame**: Body frame (robot's local coordinate system)
- **Description**: Direction of gravity vector projected into the robot's body frame
- **Range**: Each component in [-1, 1], vector magnitude = 1.0

**Index 6**: `projected_gravity[0]` = `gx` (gravity component along body X-axis)
- Positive: Gravity pulling forward (robot tilted backward)
- Negative: Gravity pulling backward (robot tilted forward)
- Source: Your accelerometer `move` (forward/backward axis)
- Calculation: `normalize([move, shift, translate])[0]`

**Index 7**: `projected_gravity[1]` = `gy` (gravity component along body Y-axis)
- Positive: Gravity pulling left (robot tilted right)
- Negative: Gravity pulling right (robot tilted left)
- Source: Your accelerometer `shift` (left/right axis)
- Calculation: `normalize([move, shift, translate])[1]`

**Index 8**: `projected_gravity[2]` = `gz` (gravity component along body Z-axis)
- Positive: Gravity pulling up (robot upside down)
- Negative: Gravity pulling down (robot upright)
- Source: Your accelerometer `translate` (up/down axis)
- Calculation: `normalize([move, shift, translate])[2]`
- When upright: Should be approximately `-1.0` (gravity down)

---

### Index 9-11: `velocity_commands` (3 values) - Desired Velocity Commands
- **Units**: m/s for linear, rad/s for angular
- **Frame**: Body frame
- **Description**: User's desired velocity command (where the robot should move)

**Index 9**: `velocity_commands[0]` = `lin_vel_x` (desired forward/backward velocity)
- Range: [-0.6, 0.6] m/s
- Positive: Move forward
- Negative: Move backward

## Critical Ordering Consistency

The following must use the **exact same joint order**:

1. `joint_pos` (observation indices 12-23)
2. `joint_vel` (observation indices 24-35)
3. `last_action` (observation indices 36-47)
4. Model action output (indices 0-11)

**Current order (`"by_type"`)**:
```
[FL_hip, FR_hip, BL_hip, BR_hip, FL_upper, FR_upper, BL_upper, BR_upper, FL_lower, FR_lower, BL_lower, BR_lower]
```

This means:
- Observation `joint_pos[0]` = FL_hip position
- Observation `joint_vel[0]` = FL_hip velocity
- Observation `last_action[0]` = previous FL_hip action
- Model output `Action[0]` = new FL_hip command

**All four must align** for correct behavior.

## Joint Ordering Schemes

### "by_type" (Current)
Order: `[FL_hip, FR_hip, BL_hip, BR_hip, FL_upper, FR_upper, BL_upper, BR_upper, FL_lower, FR_lower, BL_lower, BR_lower]`

- Groups all hips together, then all uppers, then all lowers
- Matches Anymal's convention in Isaac Lab

### "by_leg" (Alternative)
Order: `[FL_hip, FL_upper, FL_lower, FR_hip, FR_upper, FR_lower, BL_hip, BL_upper, BL_lower, BR_hip, BR_upper, BR_lower]`

- Groups all joints for each leg together
- FL complete, then FR complete, then BL complete, then BR complete

**Note**: The ordering scheme is controlled by `config.JOINT_ORDERING_SCHEME` and must be consistent across all joint-related vectors.
